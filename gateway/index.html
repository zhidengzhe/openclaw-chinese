5分钟本地启动

步骤 1：启动网关
openclaw gateway --port 18789 # debug/trace 镜像到标准输出 openclaw gateway --port 18789 --verbose # 强制终止选定端口上的监听器，然后启动 openclaw gateway --force
步骤 2：验证服务健康状态
openclaw gateway status openclaw status openclaw logs --follow

健康基准：Runtime: running 和 RPC probe: ok。

步骤 3：验证渠道就绪状态
openclaw channels status --probe

在网关可达的情况下，这会运行实时的每个账户渠道探测和可选审计。

如果网关不可达，CLI 将回退到仅配置的渠道摘要，而不是实时探测输出。

注意

网关配置重新加载监视活动配置文件路径（从配置文件/状态默认值解析，或设置 OPENCLAW_CONFIG_PATH 时）。

默认模式是 gateway.reload.mode="hybrid"。

第一次成功加载后，运行进程提供活动内存配置快照；成功的重新加载会原子性地交换该快照。

运行时模型

  • 一个始终运行的进程，用于路由、控制平面和渠道连接。
  • 单一多路复用端口用于：
    • WebSocket 控制/RPC
    • HTTP API，OpenAI 兼容（/v1/models、/v1/embeddings、/v1/chat/completions、/v1/responses、/tools/invoke）
    • 控制 UI 和钩子
  • 默认绑定模式：loopback。
  • 默认需要身份验证。共享密钥设置使用 gateway.auth.token / gateway.auth.password（或 OPENCLAW_GATEWAY_TOKEN / OPENCLAW_GATEWAY_PASSWORD），非环回反向代理设置可以使用 gateway.auth.mode: "trusted-proxy"。

OpenAI 兼容端点

OpenClaw 最高杠杆兼容性表面现在是：

  • GET /v1/models
  • GET /v1/models/{id}
  • POST /v1/embeddings
  • POST /v1/chat/completions
  • POST /v1/responses

为什么这组端点重要：

  • 大多数 Open WebUI、LobeChat 和 LibreChat 集成首先探测 /v1/models。
  • 许多 RAG 和内存管道期望 /v1/embeddings。
  • 代理原生客户端越来越倾向于 /v1/responses。

规划说明

/v1/models 是代理优先的：它返回 openclaw、openclaw/default 和 openclaw/<agentId>。

openclaw/default 是稳定别名，始终映射到配置的默认代理。

当您需要后端提供商/模型覆盖时使用 x-openclaw-model；否则所选代理的正常模型和嵌入设置保持控制。

所有这些都在主网关端口上运行，并使用与网关 HTTP API 其余部分相同的可信操作员身份验证边界。

端口和绑定优先级

设置 解析顺序
网关端口 --port → OPENCLAW_GATEWAY_PORT → gateway.port → 18789
绑定模式 CLI/覆盖 → gateway.bind → loopback

热重载模式

gateway.reload.mode 行为
off 无配置重载
hot 仅应用热安全更改
restart 在需要重新启动的更改时重新启动
hybrid（默认） 安全时热应用，否则重新启动

运维最佳实践

  • 监控日志：定期检查 openclaw logs 输出
  • 健康检查：设置定期 openclaw gateway status 检查
  • 备份配置：定期备份网关配置文件
  • 更新策略：在低流量时段执行网关更新
  • 灾难恢复：制定网关故障恢复计划

提示

对于生产环境：

  1. 使用 gateway.auth.token 进行安全身份验证
  2. 配置适当的日志级别和轮转
  3. 设置监控和告警
  4. 定期进行压力测试

故障排除

常见问题及解决方案：

  • 网关无法启动：检查端口是否被占用，权限是否足够
  • 连接失败：验证网络配置和防火墙设置
  • 性能问题：检查系统资源使用情况，调整配置参数
  • 配置错误：使用 openclaw config validate 验证配置
